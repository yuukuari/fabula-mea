Feature 1
Je voudrais ajouter un système de tickets/retour d'expérience simple sur le site, que chaque utilisateur connecté puisse faire un retour : bug, question ou amélioration un peu comme sur github.

Cela va nécessiter 3 choses :
- Une notion d'admin, avec un espace admin, qui peut voir plusieurs pages supplémentaires
- Un système de tickets
- Un système de releases

Administrateurs :
Il faut qu'un utilisateur puisse être admin (ce sera fait manuellement dans la base de données)
Un administrateur a une section Administration dans le menu, qui lui permet de voir des pages supplémentaires
- Membres : liste des membres avec infos générales et quelques statistiques
- Tickets : liste des tickets (on va le voir juste après)
- Release : liste des releases (on va le voir juste après)
Dans le menu, faire un sous-menu Adminitrateurs, sinon le menu sera trop long

Pour le système de tickets :
Une petite bulle est toujours visible en bas à gauche, un peu comme le pomodoro.
Quand on clique dessus, on peut :
- ajouter un ticket
- ou accéder à la liste des tickets
Quand on crée un ticket :
- On doit remplir type(*), titre(*), description(*), visibilité (publique ou privé, par défaut publique)
- le message peut être écrit avec tiptap mais sans l'image, juste les options de mise en page classiques
- selon le type, un template se met dans la zone de texte
- si on change le type mais qu'il y a déjà un message (différent du template par défaut), on demande confirmation avant d'écraser le texte avec le nouveau template
- Un message de confirmation apparait quand on valide, avec un lien vers la liste des tickets
- un mail est envoyé à l'adresse support client du site (variable d'environnement) avec le mail jonathancambot+eml-tickets@hotmail.com avec le détail du ticket
- Le ticket a un statut, par défaut à "Ouvert", et qui pourra être "Fermé", soit en "Terminé" soit en "Dupliqué"
La liste des tickets est une nouvelle page, où on peut voir tous les tickets, leur contenu, leur statut
- On ne peut voir que les tickets publiques + ses propres tickets
- Seul l'admin peut ajouter des commentaires sur tous les tickets
- tout le monde peut ajouter une interaction sur les commentaires (smiley)
- L'admin peut changer le statut du ticket
- Une petite ligne s'insère avec le changement de statut
- L'admin peut affecter le ticket à une release (voir système de releases)
- L'admin peut aussi supprimer le ticket
- L'admin a un lien Tickets dans le menu, qui accède à la même page de tickets

Pour le système de releases :
Une release est un ensemble de bugfix et d'améliorations qui sont livrées en une fois sur le site, avec un numéro x.y.z
Je voudrais que les utilisateurs puissent voir la release actuelle et la release note, avec les releases passées et à venir
Quand une nouvelle release sort, une modale doit les avertir (une seule fois), qui leur permet d'aller voir le détail de la release avec un bouton, ou de fermer
- Je voudrais indiquer quelle est la release actuelle sur le site en petit en bas
- Cliquer sur ce numéro de release permet de voir la liste des releases notes
- Cette nouvelle page de releases notes, accessible uniquement par ce lien, indique toutes les releases dans l'ordre et ce qu'elles apportent (titre, type, description). Elle les affiche en vertical avec une timeline et des points pour chaque release.
- Comme on peut rattacher un ticket à une release, si ce ticket est dans la release, ajouter un "Merci aux contributeurs" avec les pseudos de ceux qui ont fait les tickets
Pour l'administrateur :
- Il doit pouvoir voir la liste des releases
- créer / modifier / supprimer des releases
- Voir les tickets associés à une release
- Ajouter lui-même les bugs et améliorations de cette release avec description (c'est séparé des tickets, les tickets sont juste informatifs pour les contributeurs et indiquer dans le ticket quand il sera released)

Donc si on récapitule :
- Des utilisateurs sont admins, ils ont une section dans le menu en plus
- Ils peuvent gérer des releases
- Les utilisateurs peuvent créer des tickets, qui peuvent être intégrés à des releases






Feature 2
Il faudrait ajouter une grosse feature maintenant qu'on a la base de données : un système de relecture de son livre par d'autres personnes, avec ajout de commentaires, review des commentaires par l'auteur (échange de messages).
Je voudrais qu'on puisse faire les choses suivantes :

Ajouter un nouveau menu Relectures
Créer une demande de relecture à partager
Choisir un ou plusieurs chapitres / scènes à partager
Choisir à qui partager le contenu : à quelqu'un en copiant un lien ou en tapant des emails
On crée la demande, ou session de relecture en base avec date, contenu à relire (les chapitres et scènes sont copiées à l'instant t, pour qu'entretemps on puisse continuer d'écrire si on veut)
Le lien, crypté, contient ce qu'il faut pour voir le contenu à relire.
L'utilisateur arrive d'abord sur une page qui explique qu'il va relire tel livre de telle personne, il choisit son nom, on lui explique ce qu'on attend de lui, et que ses commentaires seront envoyés par mail à l'auteur une fois validés, et il démarre la relecture
Il voit une page similaire à celle du module d'écriture, mais cette fois-ci avec 3 volets :
Le plan, avec uniquement les chapitres et scènes qu'il doit relire
Le contenu, mais non modifiable
Une zone de commentaires
Il peut sélectionner une portion de texte, faire clic droit, ajouter un commentaire, qui apparait dans la section de droite. Un peu comme les commentaires sur un word ou google doc.
Le texte associé à un commentaire est coloré.
Quand on passe sa souris au-dessus du texte, ou du commentaire, l'élément associé est mis légèrement en avant
On peut ajouter d'autres messages à la suite d'un commentaire, le modifier, le supprimer (uniquement les siens)
Et donc voir les réponses qu'à faite l'auteur
Le commentaire a aussi un statut, en brouillon par défaut, puis envoyé, pour qu'on puisse voir les commentaires en attente de review par l'auteur, qu'il pourra clôturer
Les commentaires que fait le relecteur envoient une notification à l'auteur par mail, mais pas à chaque commentaire. Il voit ceux en brouillon, un bouton indique à tous moment en bas de la zone commentaire le nombre de commentaire à envoyer. En cliquant dessus, il les passe au statut envoyé et l'auteur reçoit un mail récap avec un lien vers la session de relecture.
Il faut aussi pouvoir indiquer quand on a totalement terminé la relecture. On peut valider des commentaires, mais continue le lendemain. Il faut cliquer sur un bouton pour dire qu'on a terminé.
Je le répète, il y a une session de relecture par lecteur. Donc si on demande à 2 personnes de relire une scène, puis le lendemain à ces 2 mêmes personnes de relire la même scène modifiée, on aura créé 2 sessions de relecture et 4 lignes apparaitront dans la page des relectures.
L'auteur voit les relectures, il peut ouvrir une vue identique à celle du lecteur. Il voit les commentaires, peut y répondre comme une discussion, et les clôturer.
Il voit dans la liste quand les relectures sont terminées



Plan : Système de relecture
C'est une feature ambitieuse mais bien découpable. Voici l'architecture complète avant de coder.

Modèle de données
ReviewSession (une par lecteur invité)

id, bookId, userId (auteur), token (UUID aléatoire public)
readerName, readerEmail?
status: 'pending' | 'in_progress' | 'completed'
snapshot: { chapters: [...], scenes: [...] }  ← copie à l'instant t
createdAt, completedAt?

ReviewComment (un par annotation)

id, sessionId, sceneId
isAuthor: boolean  ← distingue auteur vs lecteur
authorLabel: string  ← "Jonathan" ou le nom choisi par le lecteur
selectedText: string  ← le texte sélectionné
startOffset, endOffset: number  ← position en caractères dans le texte brut
content: string
status: 'draft' | 'sent' | 'closed'
parentId?: string  ← réponses en fil de discussion
createdAt, updatedAt

Clés Redis

emlb:review:{token}                    → session (token = clé d'accès public)
emlb:u:{userId}:reviews               → liste des sessionIds de l'auteur
emlb:review:{sessionId}:comments      → liste des commentaires JSON

API (Vercel Serverless)
Routes publiques (token dans l'URL, pas de JWT)

GET  /api/review/[token]              → session + snapshot
POST /api/review/[token]/start        → { readerName } → démarre la session
GET  /api/review/[token]/comments     → liste commentaires
POST /api/review/[token]/comments     → créer/modifier commentaire
DELETE /api/review/[token]/comments/[id]
POST /api/review/[token]/send         → passe les drafts en 'sent' + email à l'auteur
POST /api/review/[token]/complete     → marque la session terminée

Routes auth (auteur)

GET  /api/reviews                     → liste sessions de l'auteur
POST /api/reviews                     → créer une session (choisir scènes + destinataires)
POST /api/reviews/[id]/reply          → répondre à un commentaire
POST /api/reviews/[id]/comments/[id]/close

Token & lien
Le token est un UUID aléatoire (pas JWT) stocké en Redis. L'URL publique sera :

https://app.com/review/{token}

Avantage : révocable, pas besoin de JWT secret côté lecteur.

Emails
On a besoin d'un service d'envoi. Resend est le plus simple pour Vercel (1 package, free tier 3 000 mails/mois, API REST). Ça s'installe en 5 min. On envoie :

Email à l'auteur quand le lecteur clique "Envoyer mes commentaires" (récap des nouveaux commentaires + lien vers la session)
Optionnellement : email au lecteur avec le lien quand l'auteur crée la session
Highlighting du texte (le point délicat)
Le contenu des scènes est du HTML (TipTap). Pour ancrer un commentaire :

Quand l'utilisateur sélectionne du texte → window.getSelection() → on extrait le texte sélectionné + son offset en caractères dans le texte brut de la scène (HTML strippé)
On stocke { selectedText, startOffset, endOffset } dans le commentaire
Pour l'affichage : on reconstruit le HTML en injectant des <mark data-comment-id="xxx"> autour des ranges concernés
C'est faisable proprement avec un parser HTML léger. Pour v1, on peut simplifier : chercher le texte dans le contenu et surligner la première occurrence correspondant à l'offset.

Interface lecteur (3 panneaux)
┌──────────┬──────────────────────┬──────────────────┐
│ Plan     │ Contenu (read-only)  │ Commentaires     │
│ (chap +  │ HTML des scènes      │ Fil par scène    │
│  scènes) │ Texte sélectionnable │ + zone de saisie │
│          │ avec marques colorées│ Draft / Envoi    │
└──────────┴──────────────────────┴──────────────────┘

Barre du bas : [3 commentaires en attente d'envoi] → Envoyer / [Terminer la relecture]

Phases d'implémentation
Phase 1 — Backend (2-3h)

Types ReviewSession, ReviewComment
Toutes les routes API
Intégration Resend pour les emails
Phase 2 — Interface auteur (2h)

Page /reviews : liste des sessions (statut, lecteur, nb commentaires)
Dialogue "Nouvelle demande" : sélection chapitres/scènes + copie du lien / saisie emails
Vue auteur d'une session : même layout 3 panneaux, commentaires avec réponses + clôture
Phase 3 — Interface lecteur (3h)

Page /review/[token] : page d'accueil (titre, auteur, nom du lecteur)
Layout 3 panneaux lecture + commentaires
Barre draft/envoi/terminé
Phase 4 — Sélection de texte & highlighting (2h)

Détection de sélection, tooltip "Commenter"
Injection des <mark> dans le HTML selon les offsets
Synchronisation survol commentaire ↔ texte surligné
